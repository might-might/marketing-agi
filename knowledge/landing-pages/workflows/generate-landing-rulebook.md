# Generate Landing Page Rulebook

## Document Metadata

- **Version:** 1.0.0
- **Purpose:** Define rules for generating or updating landing pages in student projects

---

## Purpose

This rulebook defines the process for creating or updating a landing page in the `webpages` folder of a student project. The landing page must be based on validated customer needs and user instructions, without marketing clichés or personal data leakage.

---

## Step 1. Определение студенческого проекта (критично)

**Очень важно не ошибиться с выбором проекта.** Изменение чужого проекта может серьёзно навредить другому студенту.

### Способы определения целевого проекта

Используй **все доступные сигналы** и действуй по приоритету:

1. **Явно указанный путь или источник данных**
   - Пользователь указал путь к файлу: `student-projects/[student-name]/...` → проект определён
   - Пользователь сослался на файл (например, call-database.md) → путь к файлу однозначно определяет проект

2. **Открытый в IDE файл**
   - Если в редакторе открыт файл из `student-projects/[student-name]/...`, считаем этот проект целевым
   - Проверь путь открытого файла перед началом работы

3. **Контекст запроса**
   - Пользователь упомянул имя студента или проекта
   - Пользователь работал с данным проектом в предыдущих сообщениях

### Защита от ошибки

**Обязательные проверки перед любыми изменениями:**

- [ ] **Подтверждение:** Если проект не очевиден из одного источника — явно спроси пользователя: «Работаем с проектом `[student-name]`?»
- [ ] **Единственность:** Убедись, что определён ровно один проект. Если возможны два — не действуй, запроси уточнение.
- [ ] **Проверка пути:** Все пути к файлам должны начинаться с `student-projects/[student-name]/`. Никогда не пиши в `student-projects/` без имени студента.
- [ ] **Читай перед записью:** Перед изменением файла проверь, что он принадлежит выбранному проекту.

**Структура студенческих проектов:**
```
student-projects/
  [student-name]/           ← имя папки = идентификатор проекта
    webpages/               ← целевая папка для landing page
      landing.html      ← целевой файл по умолчанию (создавать, если отсутствует)
    genotype/
      call-database.md      ← источник потребностей
    customer-calls/
    employees/
```

---

## Step 2. Источники инструкций по созданию/изменению landing page

Определи, откуда брать содержание и структуру страницы.

### Вариант A: Инструкции пользователя или ссылки на файлы

- Пользователь явно описал, какие изменения нужны
- Пользователь указал файл(ы) с текстом, структурой или макетом
- **Действие:** Следуй инструкциям и используй указанные файлы как источник контента

### Вариант B: Данные из call-database.md

- Пользователь не дал явных инструкций, но в проекте есть `genotype/call-database.md`
- **Действие:**
  1. Открой `student-projects/[student-name]/genotype/call-database.md`
  2. Изучи раздел «Потребности (Needs)»
  3. Выдели самые популярные и востребованные потребности по:
     - **Частота** (количество упоминаний)
     - **Приоритет** (HIGH > MEDIUM > LOW)
     - **Source Type** (Client Direct Quote ценнее, чем Employee Statement with Client Confirmation)
  4. Используй эти потребности как основу для блоков и заголовков landing page

### Комбинация

- Если есть и инструкции пользователя, и call-database.md — объединяй: инструкции задают направление, call-database даёт валидированный контент.

---

## Step 3. Генерация или обновление landing page

### 3.0. Определение целевого файла (обязательно перед записью)

**Целевой файл по умолчанию:** `student-projects/[student-name]/webpages/landingpage.html`

1. **Проверь наличие** `webpages/landing.html` в проекте.
2. **Если `landing.html` отсутствует** → **создай** его. Не используй `index.html` или какие-то другие файлы вместо него.
3. **Если `landing.html` существует** → обновляй его.
4. **Исключение:** используй `index.html` только если пользователь **явно** попросил обновить именно index.html или указал путь к нему.

**Частая ошибка:** наличие `index.html` в `webpages/` не означает, что это целевой файл. По умолчанию цель — `landing.html`.

### Правила контента

1. **Чёткий текст без marketing-bullshit и клише**
   - Избегай: «уникальное решение», «инновационный подход», «лучший на рынке», «революционный продукт»
   - Пиши конкретно: что делает продукт/услуга, какую задачу решает, для кого

2. **Основа — валидированные потребности**
   - Текст должен отражать реальные потребности из call-database (Client Direct Quote, подтверждённые потребности)
   - Не придумывай потребности «из головы»

3. **Запрет на персональную информацию и результаты голосований**
   - Не указывай имена клиентов, компании, контакты из транскриптов
   - Не публикуй результаты опросов/голосований («7 человек проголосовали за X», «3–4 человека выбрали Y»)
   - Исключение: если пользователь **явно** просит добавить отзыв/цитату лояльного клиента — тогда можно, с его согласия

4. **Честный информационный стиль**
   - Нацеленность на массовую платёжеспособную аудиторию
   - Без манипуляций, без «заигрывания»
   - Открыто и по делу: что предлагаем, кому, зачем

### Технические требования

- Валидный HTML5
- Адаптивная вёрстка (viewport, mobile-friendly)
- Язык страницы: соответствует языку проекта (ru/en — см. call-database или инструкции)

---

## Step 4. Проверка перед сохранением

Перед записью файла убедись:

- [ ] **Целевой файл:** Записываешь в `landing.html`
- Путь к файлу содержит правильное `[student-name]`
- Нет персональных данных из звонков (можно цитаты, но нельзя указвыать чьи, только если пользователь специально не указал что создаем блок с отзывом пользователя такого-то)
- Нет результатов голосований/опросов (если пользователь не просил)
- Текст без маркетинговых клише (Купи прямо сейчас! Компания лидер рынка с командой профессионалов! Скидка сгорает через час! и т.п.)
- Контент основан на валидированных потребностях

## Output

- **Созданный или обновлённый файл:** `student-projects/[student-name]/webpages/landing.html`
- **Сообщение пользователю:** Кратко опиши, что сделано и на основе каких источников
